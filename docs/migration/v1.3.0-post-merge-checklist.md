# v1.3.0 post-merge checklist (cross-repo work)

After the rforge plugin v1.3.0 ships, the original `rforge-mcp` artifact
should be retired. This page documents what that meant in practice and
what was actually done.

## Scope correction (2026-05-11)

The original SPEC assumed `rforge-mcp` lived as a public GitHub repo at
`data-wise/rforge-mcp` and on npm as `rforge-mcp`. **Neither was ever
true.** The prototype only existed as a local working directory at
`~/projects/dev-tools/mcp-servers/rforge` on the author's machine,
exposed globally via `npm link` (not `npm publish`).

So the checklist was rewritten from "archive the GitHub repo + push
DEPRECATED.md" to the actual cleanup tasks below.

## 1. Drop the global `npm link` symlink

```bash
npm uninstall -g rforge-mcp
```

This removes the symlink at `$(npm prefix -g)/lib/node_modules/rforge-mcp`
that pointed at the local source. After this, `npm list -g` no longer
shows rforge-mcp, and any `node /opt/homebrew/bin/rforge-mcp` invocation
fails (which is what we want — the plugin's `lib/` modules are now the
intended surface).

## 2. Tombstone the local source dir

```bash
# In the local rforge-mcp source dir
cp ~/projects/dev-tools/rforge/docs/migration/rforge-mcp-deprecation.md \
   ~/projects/dev-tools/mcp-servers/rforge/DEPRECATED.md
```

Optionally rename the directory itself to make its retired state obvious
in `ls` output:

```bash
mv ~/projects/dev-tools/mcp-servers/rforge \
   ~/projects/dev-tools/mcp-servers/rforge-archived
```

## 3. Update your own `~/.claude/settings.json`

If your personal `~/.claude/settings.json` has an `mcpServers.rforge`
entry, remove it manually. We don't auto-edit user settings. Example:

```json
// Before:
"mcpServers": {
  "rforge": { "command": "...", "args": [...] },
  "other-server": { ... }
}

// After:
"mcpServers": {
  "other-server": { ... }
}
```

## 4. (Skipped — does not apply)

- **No GitHub repo to archive** — `data-wise/rforge-mcp` never existed
  publicly. `gh repo view` returns `Could not resolve to a Repository`.
- **No npm package to deprecate** — `npm view rforge-mcp` returns 404.
- **No Homebrew formula** — never had one.

## Verification

After completing steps 1 and 2:

- `npm list -g rforge-mcp` returns nothing (was: `rforge-mcp@0.1.0 → ...`)
- `which rforge-mcp` returns nothing (binary is no longer on PATH)
- The local source dir is tombstoned with `DEPRECATED.md` and/or renamed

## Rollback

If you ever need to resurrect rforge-mcp:

- The local source remains at `~/projects/dev-tools/mcp-servers/rforge-archived/`
  (or `rforge/` if you skipped the rename).
- Re-link it globally with `cd <source-dir> && npm link`.
- Restore the `mcpServers.rforge` entry in `~/.claude/settings.json`.

Note: as of v1.3.0, the plugin's pure-Python `lib/` modules cover every
implemented tool that rforge-mcp shipped — there is no functional reason
to restore the MCP server. This rollback path is a safety net, not a
recommended workflow.
