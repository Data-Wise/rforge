# Hooks & Skills Reference

> Added in v1.2.0. Mirrors craft's `.claude-plugin/` layout.

RForge ships two pieces of autonomous tooling that activate without an
explicit `/rforge:` command:

- **Hooks** ‚Äî fire on every Claude Code tool call (e.g. `Write`, `Edit`).
- **Skills** ‚Äî autonomous validators discoverable by Claude when it needs
  to verify package state.

This page is the user-facing reference for what's currently shipped.

---

## Hooks

### `pretooluse.py`

A **PreToolUse** hook that runs on every `Write` and `Edit` call. It
applies four R-package-aware rules. Only one of them blocks; the rest
print a warning to stderr and let the operation proceed (matches craft's
"warn loudly, block rarely" philosophy).

| # | Rule | Severity | Trigger |
|---|------|----------|---------|
| 1 | Block edits to roxygen-generated `.Rd` files | **BLOCK** (exit 2) | `file_path` matches `*/man/*.Rd` |
| 2 | Remind on `R/*.R` edits | warn | `file_path` matches `*/R/*.R` |
| 3 | SemVer check on `DESCRIPTION` Version | warn | `file_path` ends in `DESCRIPTION` and `Version:` line is present |
| 4 | Outside-worktree write | warn | CWD is a git worktree, but `file_path` resolves outside it |

#### Rule 1 ‚Äî Block edits to `man/*.Rd`

Hand-edits to `man/*.Rd` files are always wrong: those files are
regenerated by `devtools::document()` from roxygen comments in `R/`. Any
manual change is silently destroyed on the next regeneration. The hook
blocks the call (exit 2) and prints:

```text
üõë Refusing to edit roxygen-generated man/*.Rd file
   File:    /path/to/man/foo.Rd
   Reason:  man/*.Rd is regenerated by devtools::document().
   Instead: edit the @param/@return/@examples roxygen comments in R/
            and run devtools::document() to regenerate.
```

To regenerate properly, edit the corresponding `R/foo.R` and run:

```r
devtools::document()
```

#### Rule 2 ‚Äî Remind on `R/*.R` edits

Editing R source can require keeping `NAMESPACE` and `DESCRIPTION` in
sync ‚Äî adding `@export` tags, new dependencies, etc. The hook can't
*verify* this (it can't see future edits), so it just reminds:

```text
‚öÝÔ∏è  Editing R/ source ‚Äî remember to keep package metadata in sync
   File: /path/to/R/foo.R
   If you added/removed @export tags: run devtools::document().
   If you added a new dependency: add it to DESCRIPTION (Imports:).
```

#### Rule 3 ‚Äî SemVer check on `DESCRIPTION` Version

When writing or editing `DESCRIPTION`, the hook scans for a `Version:`
line and verifies it matches `MAJOR.MINOR.PATCH` (with optional pre-release
suffix). If it doesn't, you get:

```text
‚öÝÔ∏è  DESCRIPTION Version 'v1.2' is not SemVer-compatible
   Expected:  MAJOR.MINOR.PATCH (e.g. 1.2.0 or 1.2.0-rc1)
   CRAN policy: prefers strict X.Y.Z bumps for releases.
```

CRAN itself accepts a wider variety of version strings, but strict SemVer
makes downstream packaging (Homebrew, npm, etc.) much simpler.

#### Rule 4 ‚Äî Outside-worktree write

Direct port of craft's `pretooluse.py` rule. When you're working in a
git worktree under `~/.git-worktrees/`, writes to files outside the
worktree usually mean a wrong-CWD bug. The hook warns:

```text
‚öÝÔ∏è  Writing outside worktree
   File:     /Users/you/projects/other-repo/foo.txt
   Worktree: /Users/you/.git-worktrees/rforge/feature-x
   Consider: cd /Users/you/.git-worktrees/rforge/feature-x
```

### Wiring up the hook

The hook is **not auto-wired** by simply existing in the plugin. To
activate, add to your `~/.claude/settings.json` (or per-project
`.claude/settings.json`):

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/.claude-plugin/hooks/pretooluse.py"
          }
        ]
      }
    ]
  }
}
```

### Testing the hook locally

Each rule can be exercised by piping JSON to stdin (Claude Code's hook
contract ‚Äî see `.claude-plugin/hooks/README.md` for details):

```bash
# Block: editing a roxygen-generated .Rd file (exit 2)
echo '{"tool_name":"Edit","tool_input":{"file_path":"man/foo.Rd","old_string":"x","new_string":"y"}}' \
  | python3 .claude-plugin/hooks/pretooluse.py
echo $?

# Warn: bad SemVer in DESCRIPTION (exit 0)
echo '{"tool_name":"Write","tool_input":{"file_path":"DESCRIPTION","content":"Version: not-semver\n"}}' \
  | python3 .claude-plugin/hooks/pretooluse.py
echo $?
```

## Skills

Skills live under `.claude-plugin/skills/<category>/` and are
auto-discoverable by Claude based on their YAML frontmatter. Each skill
is a single Markdown file.

### `validation/description-sync.md`

Verifies that `DESCRIPTION` `Version:` matches the top entry of
`NEWS.md` or `CHANGELOG.md` ‚Äî the most common pre-release failure mode
in R packages. Pure shell, no R required.

#### Modes

| Mode | Action on mismatch |
|------|--------------------|
| `default` | Warn, exit 0 |
| `debug` | Verbose diff, exit 0 |
| `optimize` | Warn, exit 0 (no auto-fix ‚Äî version bumps are deliberate) |
| `release` | Fail, exit 1 |

#### Example output

```text
‚úÖ PASS: DESCRIPTION Version 1.2.0 found in NEWS.md
```

‚Ä¶or, if drift is detected in `release` mode:

```text
‚ùå FAIL: DESCRIPTION Version 1.2.0 is not in NEWS.md (top 50 lines)
   Add an entry to NEWS.md before tagging the release.
```

#### When this runs

- Manually, by extracting the script block and running it.
- Auto-discovered by `/rforge:check` once Phase 4 (discovery engine) lands.
- As a CRAN-prep sanity step.
